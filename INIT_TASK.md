# 任务书：png2vsdx 核心执行层架构调研与实施计划

## 角色设定
你是 `png2vsdx` 项目的核心开发 Agent。你的目标是构建一个健壮、可扩展的基于 Visio 的多模态排版系统。请严格遵循“先调研、后汇报、再开发”的原则。

## 核心任务与环境约束
我们目前处于**阶段一**：构建 Agent 到 Visio 的操作链路（即“执行手”）。
- **执行环境约束**：你（Agent 本体）运行在 WSL (Linux) 环境中。
- **目标操作软件**：Microsoft Visio 运行在宿主机的 Windows 环境中。
- **版本控制**：全程需要使用 Git 并同步到 GitHub 远程仓库。

## 核心架构设计要求
你需要围绕以下两个核心目录设计并实现控制逻辑：
1. `visioskills/`（原子操作库）：封装与 Visio 交互的最基础动作（如：画框、连线、改色）。这部分需要通过 MCP (Model Context Protocol) 暴露给你自己。
2. `drawskills/`（复合技能库）：存放拓扑记忆和审美规范（如：如何画一个 ResNet 块，IEEE 论文规范配色等），供你调用组合。

## 调研与分析目标 (Action Required)
在编写任何具体业务代码之前，你需要完成以下调研并向主管（User）提交一份《技术实施方案》：

1. **跨系统通信方案**：如何在 WSL 中安全、稳定地调用 Windows 宿主机的 Visio COM 接口？（提示：考虑在 Windows 端运行 Python Flask/FastAPI 封装的 MCP Server，WSL 通过 HTTP/TCP 调用；或探讨其他跨 OS 的 MCP 桥接方案）。
2. **MCP 工具定义**：初步列出 `visioskills` 中需要实现的头 5 个基础操作 API 及其参数结构。
3. **技能库存储结构**：定义 `drawskills` 中配置文件的格式（JSON/YAML），并给出一个“画两个带箭头的方框”的结构化示例。
4. **Git 工作流**：说明你将如何处理代码的 commit 和 push。

## 交付规范
1. 阅读本阶段任务后，生成一份详细的《技术实施方案》。
2. **在主管明确回复“方案通过，开始开发”之前，绝不要编写核心业务代码或创建复杂项目结构。**
3. 等待主管指令。
